Section 01 — Le modèle mental

Les quatre piliers

Quand on parle d'« infrastructure », on mélange souvent des choses très différentes : des serveurs physiques, des machines virtuelles, des bases de données, du réseau, du stockage, des services managés, des plateformes internes… Pour ne pas s'y perdre, il est utile de retenir qu'on peut ramener l'essentiel à quatre familles de ressources qui se combinent sans cesse : compute, storage, network, runtime.

Figure 1Les quatre piliers — tout service cloud se ramène à une combinaison de compute, storage, network et runtime

Tout ce qui vient après — conteneurs, Kubernetes, bases de données, plateformes internes — n'est au fond qu'une orchestration de ces quatre piliers à différents niveaux d'abstraction. Un cluster Kubernetes consomme du compute (les nœuds), du stockage (persistent volumes), du réseau (CNI, services), et offre un runtime (les pods). Une base PostgreSQL managée consomme elle aussi compute+stockage+réseau, mais te l'expose comme un simple endpoint.

Les sections suivantes entrent dans chacun de ces piliers, et terminent par la couche d'orchestration et les disciplines transversales — platform engineering, FinOps, Green IT.

Section 02 — Calcul

Le compute en profondeur

Quand tu veux faire tourner du code, quatre modèles principaux s'offrent à toi aujourd'hui. Ils diffèrent dans ce que tu gères, la rapidité de démarrage, la granularité de facturation, et l'isolation.

Figure 2Les 4 modèles de compute — de la maîtrise totale à la pure abstraction

Bare metal

Un serveur physique dédié. Tu as toute la machine pour toi, avec ses CPU, sa RAM, ses disques. Aucune couche de virtualisation. C'est le modèle le plus performant (pas de surcoût d'hyperviseur) et le plus rigide. On l'utilise aujourd'hui surtout pour des charges très spécifiques : bases de données à latence critique, calcul scientifique, entraînement de gros modèles d'IA (les GPU y sont alors directement accessibles), ou conformité règlementaire imposant une isolation physique.

Fournisseurs cloud : AWS Bare Metal, OVH Dedicated Server, Equinix Metal, Scaleway Elastic Metal.

Machines virtuelles

La VM est historiquement le socle du cloud. Un logiciel appelé hyperviseur partage un serveur physique en plusieurs machines virtuelles isolées, chacune avec son propre OS. Les hyperviseurs les plus répandus sont KVM (open source, coeur de la plupart des clouds publics et d'OpenStack), VMware ESXi (historique en entreprise), Microsoft Hyper-V et Xen. On distingue parfois les hyperviseurs de type 1 (installés directement sur le matériel) et de type 2 (tournant sur un OS hôte classique).

Chez les hyperscalers, chaque VM a un nom commercial (AWS EC2, Azure VMs, Google Compute Engine). Dans OpenStack, c'est le service Nova qui orchestre les VMs — et Orange est historiquement un gros utilisateur et contributeur OpenStack.

Containers

Le container, abordé au Volume 2, est un package applicatif qui partage le noyau de l'hôte plutôt qu'embarquer son propre OS. C'est cette différence qui rend les containers beaucoup plus légers et rapides à démarrer que les VM. Sous le capot, trois composants :

Le format
OCI Image
Spécification ouverte (Open Container Initiative) qui définit le format d'image d'un container. Docker l'implémente, mais Podman, Buildah, Kaniko aussi. On y trouve le système de fichiers figé de l'application et ses métadonnées.
Le moteur d'exécution
Container runtime
containerd et CRI-O sont les deux runtimes standards aujourd'hui (Docker utilise containerd en interne depuis 2020). Ils pilotent runc, l'exécuteur bas niveau qui utilise les primitives Linux (namespaces, cgroups) pour isoler le container.
L'orchestrateur
Kubernetes & alternatives
K8s domine. En alternative : Nomad (HashiCorp, plus simple, polyvalent VM/container), Docker Swarm (en déclin), et les services managés ECS/Fargate d'AWS. En télécom, OpenShift (Red Hat) est très présent — notamment chez Orange pour leurs fonctions réseau.

Serverless / FaaS

Le niveau d'abstraction maximal. Tu fournis une fonction — quelques dizaines ou centaines de lignes de code — et le provider la fait tourner à la demande. Pas de serveur à gérer, pas d'OS, pas de scaling. Tu es facturé à la milliseconde d'exécution. Les leaders : AWS Lambda, Azure Functions, Google Cloud Functions, Cloudflare Workers (edge serverless), Vercel Functions.

Variante importante : les conteneurs serverless comme AWS Fargate, Google Cloud Run, Azure Container Apps. Tu fournis une image Docker, le provider gère tout le reste avec une facturation à l'usage. C'est un modèle hybride très populaire pour les APIs et microservices.

Le cas du GPU compute

Avec l'explosion de l'IA générative, le GPU est devenu un pilier à part entière de l'infrastructure. Les workloads d'entraînement et d'inférence réclament des cartes spécialisées (NVIDIA H100, H200, B200 en 2026, ou plus accessibles A100, L40S pour l'inférence). Les clouds publics exposent ces cartes via des instances dédiées (AWS P5, Azure ND-series, GCP A3). Côté Kubernetes, le NVIDIA GPU Operator et le Kubeflow stack rendent l'orchestration GPU gérable.

Règle pratique

Le bon modèle de compute dépend du trafic et de l'état de ton application. Trafic constant et prévisible ? VM ou container sur Kubernetes. Trafic en pics ponctuels ? Serverless. Stateful, latence critique, conformité ? Bare metal ou VM. Le piège classique est d'utiliser le serverless pour tout — le coût au repos est nul, mais sous forte charge continue, ça coûte plus cher qu'une VM bien dimensionnée.

Section 03 — Données au repos

Le stockage — trois familles

Le stockage dans le cloud se décline en trois grandes familles, qui répondent à des besoins très différents. Les confondre est l'une des erreurs les plus coûteuses qu'on puisse faire en concevant une architecture.

Figure 3Les trois familles de stockage — chacune avec son cas d'usage canonique

Stockage bloc

Un volume de stockage bloc se comporte comme un disque dur : une séquence de blocs bruts numérotés, que tu formates avec un système de fichiers (ext4, xfs, NTFS) et attaches à une seule VM à la fois. C'est le stockage de choix pour les disques système et les bases de données.

Caractéristiques clés : latence très basse (de l'ordre de la milliseconde sur du SSD), IOPS élevés, attachement 1:1 à une instance, prix relativement élevé au Go. Les exemples canoniques : AWS EBS, Azure Managed Disks, GCP Persistent Disk, OpenStack Cinder.

Stockage fichier

Un système de fichiers partagé, accessible simultanément par plusieurs machines via des protocoles standards (NFS pour Linux, SMB/CIFS pour Windows). Organisation hiérarchique classique (dossiers, fichiers, permissions POSIX), ce qui permet à des applications legacy de travailler sans modification.

On l'utilise pour : partage de fichiers entre équipes, répertoires utilisateurs, workflows de traitement d'images ou de médias, environnements de rendu. Exemples : AWS EFS, Azure Files, GCP Filestore, OpenStack Manila, NetApp ONTAP, Dell PowerScale.

Stockage objet

Le plus récent des trois paradigmes, popularisé par S3 en 2006. Tu stockes des objets (des fichiers avec des métadonnées) dans des buckets, adressables par une URL HTTP. Pas de hiérarchie native, pas de modification in-place : pour changer un objet, tu remplaces sa version complète.

C'est le stockage qui monte à l'échelle infiniment (exabytes), avec une durabilité très élevée (11 neufs chez AWS), pour un coût dérisoire au Go — en contrepartie d'une latence plus élevée qu'en bloc. L'usage typique : backups, data lakes, images et vidéos, logs archivés, contenus statiques d'un site web, artefacts CI/CD.

Les grands de la famille : AWS S3 (le standard de fait, dont l'API est imitée partout), Azure Blob Storage, GCP Cloud Storage, OpenStack Swift, MinIO (implémentation S3-compatible open source), Backblaze B2, Wasabi (alternatives low-cost).

Les classes de stockage

Sur l'objet, chaque fournisseur propose plusieurs classes de stockage selon la fréquence d'accès :

ClasseLatenceCoûtCas d'usage
Hot / Standard ms Élevé Contenu actif, site web, pipeline actif
Infrequent Access ms Moyen Backups récents, logs des 90 derniers jours
Archive / Glacier minutes à heures Très bas Archives légales, backups anciens
Deep Archive jusqu'à 12 h Dérisoire Conformité 7-10 ans, médias rarement sollicités
Section 04 — Connectivité

Le réseau cloud

Le réseau est le pilier le plus sous-estimé des quatre. Dans le cloud, il est presque entièrement logiciel (software-defined) — ce qui permet de définir tout un réseau d'entreprise complexe avec quelques lignes de Terraform, mais aussi de tout casser très efficacement si on ne maîtrise pas les concepts.

Les concepts fondamentaux

Le réseau privé virtuel
VPC
Virtual Private Cloud. Un espace réseau logiquement isolé chez un fournisseur cloud. Tu y définis des plages d'adresses IP (ex : 10.0.0.0/16), des sous-réseaux, des règles de routage. Chez OpenStack, c'est le service Neutron qui fournit cette couche.
Segmentation
Subnet
Un sous-ensemble d'un VPC, typiquement lié à une zone de disponibilité. On sépare classiquement en subnets publics (accessibles depuis Internet) et privés (isolés, pour les bases de données et backends).
Pare-feu logiciel
Security Group / NACL
Filtres statefull (Security Group) ou stateless (NACL) qui contrôlent le trafic entrant et sortant des ressources. Tu autorises explicitement : « cette VM accepte du port 443 depuis ce security group ».
Répartition de charge
Load Balancer
Distribue le trafic entrant sur plusieurs instances. L4 (TCP/UDP, rapide, simple) vs L7 (HTTP, plus riche, routage par URL). Exemples : AWS ELB/ALB/NLB, HAProxy, Nginx, Traefik, Envoy.
Résolution de noms
DNS
Transforme les noms de domaines en IPs. Dans le cloud : Route 53, Azure DNS, Cloud DNS. Plus que de la simple résolution, le DNS moderne fait du health check, du failover, et du geo-routing.
Cache edge
CDN
Content Delivery Network. Réseau de caches distribués géographiquement près des utilisateurs. Servent des copies statiques pour réduire latence et charge origine. Cloudflare, AWS CloudFront, Akamai, Fastly.
Porte d'entrée API
API Gateway
Point d'entrée unique pour les APIs. Gère l'authentification, le rate limiting, la journalisation, la transformation des requêtes. Exemples : AWS API Gateway, Kong, Tyk, Apigee, Azure API Management.
Communication inter-services
Service Mesh
Couche de gestion de trafic, sécurité (mTLS), et observabilité entre microservices. Déploie un sidecar (ex : Envoy) à côté de chaque service. Standards du marché : Istio, Linkerd, Consul Connect.
Figure 4L'architecture type d'un VPC — subnets publics/privés, LB, NAT, et services backend

Régions, zones, edge

Une région est une zone géographique avec plusieurs datacenters (ex : eu-west-3 = Paris chez AWS). Une zone de disponibilité (AZ) est un datacenter (ou groupe de datacenters proches) à l'intérieur d'une région. Pour la haute disponibilité, on répartit les services sur plusieurs AZs : si l'une tombe, les autres prennent le relais.

L'edge computing pousse la logique encore plus près des utilisateurs : Cloudflare Workers, Fastly Compute@Edge, AWS Lambda@Edge exécutent du code dans des centaines de PoPs (Points of Presence) mondiaux. Latence de l'ordre de quelques ms pour l'utilisateur final, au prix de contraintes d'exécution plus strictes.

L'ordre de grandeur

Une bonne heuristique pour dimensionner la résilience : un seul datacenter tombe quelques heures par an, toute une région tombe quelques fois par décennie (mais ça arrive — us-east-1 d'AWS a déjà eu des incidents majeurs). Pour une app critique, multi-AZ est la base. Pour une app vitale (banque, urgence, télécom), il faut du multi-région.

Section 05 — Données structurées

Les bases de données

L'écosystème des bases de données s'est spectaculairement diversifié depuis 2005. Là où on utilisait autrefois « la base SQL » pour tout faire, on choisit aujourd'hui la bonne technologie pour chaque cas d'usage. On parle de polyglot persistence. Voici la taxonomie à avoir en tête.

SQL vs NoSQL — le vrai clivage

Contrairement à la caricature, la différence n'est pas « avec ou sans SQL ». C'est une histoire de modèle de données et de garanties. Les bases SQL (relationnelles) offrent de fortes garanties transactionnelles (ACID : atomicité, cohérence, isolation, durabilité) et un modèle tabulaire rigide. Les bases NoSQL sacrifient une partie de ces garanties pour gagner en flexibilité de modèle et en scalabilité horizontale.

Les bases SQL / relationnelles

Open source · le défaut
PostgreSQL
Le couteau suisse moderne. Extensible (PostGIS pour le géospatial, pgvector pour l'IA), ACID strict, JSONB pour du semi-structuré. Choix par défaut pour la majorité des nouveaux projets depuis 2020.
Open source · historique
MySQL / MariaDB
Omniprésent dans le web (historiquement PHP/LAMP). MariaDB est un fork communautaire de MySQL. Excellent pour les workloads standards de lecture/écriture.
Commercial · entreprise
Oracle Database
Historique, très présent dans la banque, le secteur public, les grandes entreprises. Performances élevées, prix à l'avenant. En déclin d'adoption dans les nouveaux projets.
Commercial · Microsoft
SQL Server
L'équivalent Microsoft. Très répandu dans les écosystèmes .NET, les entreprises historiquement Windows. Disponible sur Linux depuis 2017.
Distribuée moderne
NewSQL (CockroachDB, Yugabyte)
SQL traditionnel mais avec scaling horizontal natif et multi-région. Garanties ACID conservées. Alternatives récentes pour les besoins de scalabilité globale.
Managé par hyperscaler
Aurora / Cloud SQL / Azure SQL
Versions cloud-native de PostgreSQL/MySQL par AWS, GCP, Azure. Backup auto, replication, scaling simplifié. Beaucoup plus cher mais beaucoup moins d'opérationnel à gérer.

Les bases NoSQL — quatre familles

Famille 1 · Document
Bases document
Stockent des documents JSON (ou BSON) avec schéma flexible. Idéal pour les données semi-structurées et évolutives. Leader : MongoDB. Alternatives : Couchbase, Amazon DocumentDB, Azure Cosmos DB.
Famille 2 · Clé-valeur
Key-value stores
Simples mais ultra-rapides. Stockent une valeur associée à une clé. Usage : cache applicatif, session store, file d'attente simple, compteurs. Leader open source : Redis (et sa branche Valkey). Alternatives : Memcached, DynamoDB (AWS), Cassandra en mode simplifié.
Famille 3 · Column-family
Wide column
Conçues pour des volumes gigantesques et des écritures massives. Issue du Bigtable de Google. Leader : Cassandra (open source). Autres : ScyllaDB (Cassandra compatible en C++), HBase, Bigtable, DynamoDB.
Famille 4 · Graph
Bases graphes
Stockent nœuds et relations. Excellentes pour les réseaux sociaux, détection de fraude, recommandation, knowledge graphs. Leader : Neo4j. Autres : Amazon Neptune, ArangoDB, JanusGraph.

Les bases spécialisées

Séries temporelles
TSDB
Optimisées pour les données horodatées : métriques, télémétrie IoT, finance. InfluxDB, TimescaleDB, VictoriaMetrics, Prometheus. Le Volume 2 en parlait pour la supervision.
Recherche
Moteurs de recherche
Indexation plein texte, recherche pertinente, agrégations analytiques. Elasticsearch / OpenSearch, Meilisearch, Typesense, Algolia (SaaS).
Vectoriel (IA)
Vector databases
Indexent des embeddings pour la recherche sémantique et le RAG. Essentielles dans les architectures IA. Pinecone, Weaviate, Qdrant, Milvus, pgvector (extension PostgreSQL).
Analytiques
Entrepôts de données
OLAP — optimisées pour les requêtes analytiques sur des volumes massifs. Snowflake, Databricks, BigQuery, Redshift, ClickHouse (open source, montée fulgurante).
OLTP vs OLAP — vocabulaire clé

Deux mondes différents. OLTP (Online Transaction Processing)  : beaucoup de petites transactions (lire/écrire une ligne). Optimisé par les bases classiques (PostgreSQL, MySQL). OLAP (Online Analytical Processing)  : peu de requêtes, mais énormes (agrégation sur des milliards de lignes). Optimisé par les entrepôts (Snowflake, BigQuery, ClickHouse). Les confondre coûte cher : un OLAP sur MySQL rampe, un OLTP sur Snowflake est hors de prix.

Section 06 — Flux asynchrones

Messaging & streaming

Quand les services deviennent nombreux, la communication synchrone directe (service A appelle service B par HTTP) devient rapidement un cauchemar : couplage fort, pannes en cascade, lenteurs qui se propagent. La réponse : des middlewares de messagerie qui permettent aux services de communiquer de façon asynchrone, via des files ou des flux.

Deux grandes familles

Paradigme 1
Message broker (queue)
Les messages sont consommés puis disparaissent. Un consommateur prend un message dans la queue, le traite, puis il est supprimé. Idéal pour des tâches à exécuter une seule fois (envoyer un email, traiter un paiement). Leaders : RabbitMQ, ActiveMQ, AWS SQS.
Paradigme 2
Event streaming
Les messages sont conservés dans un log ordonné. Plusieurs consommateurs peuvent les lire indépendamment, à leur rythme. Les messages restent disponibles pour replay. Idéal pour l'analytique temps réel, l'event sourcing, l'alimentation de plusieurs systèmes. Leader : Kafka. Alternatives : Apache Pulsar, AWS Kinesis, Redpanda (Kafka compatible, plus rapide).
Figure 5Queue vs Stream — le bon outil selon que tu traites des tâches ou des événements

Dans l'univers cloud public

Chaque hyperscaler propose ses services managés équivalents :

BesoinAWSGCPAzure

Le piège classique

Kafka est devenu presque un buzz-word. Beaucoup d'équipes le déploient sans réellement avoir besoin de ses caractéristiques (rétention longue, scaling horizontal, multi-consumer). Si tu as juste besoin d'envoyer des emails en asynchrone, RabbitMQ ou même AWS SQS est beaucoup plus simple à opérer qu'un cluster Kafka à 5 nœuds avec ZooKeeper (ou KRaft), registre de schémas et Schema Registry.

Section 07 — Stratégies cloud

Les modèles de cloud

Il n'existe pas un cloud. Il existe plusieurs stratégies de déploiement qui répondent à des contraintes différentes — souvent un compromis entre coût, agilité, souveraineté, conformité et autonomie.

Le modèle phare
Public cloud
Infrastructure opérée par un tiers (AWS, Azure, GCP, OVH) mutualisée entre ses clients. Pay-as-you-go, scalabilité quasi-infinie. Le plus agile, le moins cher à petite échelle. Les hyperscalers dominent : AWS (~30 % du marché), Azure (~25 %), GCP (~12 %).
Maîtrise totale
Private cloud
Infrastructure cloud-like (API, self-service, facturation) mais déployée sur les propres datacenters de l'organisation. OpenStack, VMware Cloud Foundation, Red Hat OpenShift, Azure Stack.
Le mariage
Hybrid cloud
Combinaison cohérente de public et privé, avec des workloads qui circulent entre les deux. Typiquement : les données sensibles sur private, les services utilisateurs sur public.
Diversification
Multi-cloud
Utilisation délibérée de plusieurs providers publics pour différentes raisons : éviter le vendor lock-in, optimiser les coûts, tirer parti des forces de chacun, ou satisfaire des contraintes règlementaires.
Souveraineté
Sovereign cloud
Cloud opéré sous droit local (ex : européen), hors d'atteinte de lois extraterritoriales comme le CLOUD Act ou le Patriot Act américains. Particulièrement pertinent pour le secteur public, la santé, la défense. Initiatives en France : Bleu (Orange + Capgemini + Microsoft), S3ns (Thales + Google), OVH SecNumCloud.
Proximité
Edge cloud
Déploiement de ressources cloud au plus près des utilisateurs et des objets connectés. Latence ultra-faible. Cas d'usage : 5G, véhicule autonome, IoT industriel, gaming. Orange et les télécoms sont naturellement bien placés pour l'edge grâce à leur maillage territorial.

Le cas Gaia-X et la souveraineté européenne

Gaia-X est une initiative lancée en 2019 par la France et l'Allemagne visant à fédérer des offres cloud européennes autour de standards d'interopérabilité, de portabilité et de transparence. L'objectif n'est pas de créer un « cloud européen » mais un cadre de labellisation permettant aux entreprises européennes de choisir des fournisseurs conformes à certains critères de souveraineté.

Orange Business est l'un des membres fondateurs de Gaia-X. Les débats sur ce projet ont été nombreux — certains estiment qu'il a dilué ses ambitions en acceptant l'adhésion des hyperscalers américains. Le sujet reste en mouvement en 2026.

Le vrai coût du multi-cloud

Le multi-cloud sonne bien en réunion stratégique, mais il multiplie la complexité opérationnelle. Chaque provider a ses APIs, ses quirks, sa tarification, ses outils. Pour qu'il soit réellement rentable, il faut soit : (1) des workloads très découplés qu'on peut vraiment bouger, (2) une équipe platform engineering solide pour abstraire les différences, (3) une volonté délibérée de négocier les prix entre fournisseurs. Sinon, c'est souvent 2× les coûts pour 0 % de bénéfices.

Section 08 — Abstraction pour les devs

Platform Engineering

La tendance majeure des années 2023-2026 dans le monde des grandes organisations. Après que le DevOps a voulu que chaque équipe maîtrise sa pile complète, on a réalisé que la charge cognitive imposée aux développeurs devenait ingérable. La solution : une équipe plateforme qui construit et maintient une plateforme interne pour développeurs (Internal Developer Platform, IDP), consommée en self-service par les équipes produit.

Le concept de golden path

Au cœur de la platform engineering, l'idée de golden path (chemin doré) : un parcours pré-configuré, opinion-né, qui permet de passer de « j'ai une idée » à « mon service tourne en prod » en quelques clics. Pas le seul chemin possible, mais le plus confortable et celui qui coche toutes les cases (sécurité, conformité, observabilité, coût). Ceux qui sortent du chemin doré le font en connaissance de cause.

Les composants d'une IDP

Le portail
Backstage
Créé par Spotify, open source CNCF. Portail unifié service catalog + templates + documentation + plugins. Le standard de fait des IDP DIY ambitieuses.
Portail commercial
Port · Cortex · OpsLevel
Alternatives SaaS plus rapides à mettre en place que Backstage. Pensées pour les équipes plateforme qui n'ont pas la bande passante pour construire leur portail.
Orchestration
Humanitec · Crossplane
Moteur qui interprète les demandes des développeurs et provisionne les ressources correspondantes dans le cloud. Crossplane est open source et très populaire en 2026.
Self-service infra
Terraform Cloud · Atlantis
Exécution pilotée de Terraform depuis des pull requests ou depuis une interface web, avec approbation, politique de sécurité, inventaire.
Self-service déploiement
ArgoCD · Flux
GitOps : les déploiements Kubernetes sont pilotés depuis un dépôt Git. L'équipe plateforme fournit les pipelines et les règles, les équipes produit fournissent leurs manifestes.
Observabilité unifiée
Stack centralisée
Metrics, logs, traces unifiés dans un seul endroit (Grafana Cloud, Datadog, ou une stack LGTM managée par l'équipe plateforme). Les équipes produit consomment sans avoir à assembler leur stack.
La règle d'or

La platform engineering fonctionne quand l'équipe plateforme traite sa plateforme comme un produit, pas comme un service interne. Ça veut dire : des utilisateurs (les devs), un backlog, des sprints, du user research, des métriques de succès (time to first deploy, taux d'adoption des golden paths, NPS interne). Une plateforme imposée sans cet état d'esprit se fait contourner en quelques mois.

Section 09 — Coûts & sobriété

FinOps & Green IT

Deux disciplines relativement récentes qui répondent à une prise de conscience tardive : les factures cloud grimpent vite, et le numérique consomme beaucoup d'énergie. Les deux se rejoignent : une infrastructure optimisée coûte moins cher et émet moins de CO₂.

FinOps — la discipline du coût cloud

Le FinOps (contraction de « Finance + DevOps ») est une pratique qui vise à donner aux équipes techniques la visibilité et les outils pour optimiser leur consommation cloud en continu. Ses principes : tout le monde est responsable du coût (pas seulement la finance), les décisions sont guidées par la donnée, et l'optimisation est un processus continu.

Étape 1 · Visibilité
Tagging & allocation
Chaque ressource cloud doit être taguée (équipe, projet, environnement). Sans tags, impossible de savoir qui consomme quoi. C'est le prérequis de tout le FinOps.
Étape 2 · Optimisation
Rightsizing & cleanup
Détecter les VMs surdimensionnées, les volumes non attachés, les snapshots oubliés, les environnements de dev qui tournent la nuit. Les outils automatisés (Kubecost pour K8s, Finout, Vantage, CloudHealth) identifient ces gaspillages.
Étape 3 · Engagement
Reserved Instances & Savings Plans
Réduire le prix en s'engageant sur 1 ou 3 ans. Économies typiques de 30 à 60 % sur les workloads stables. Piège : s'engager sur du capacity qu'on n'utilise finalement pas.
Étape 4 · Anticipation
Shift-left cost
Estimer le coût avant le déploiement, pas après. Outils : Infracost (intégré à Terraform dans les PR), Cycloid. Le dev voit combien sa PR va coûter avant de merger.

Green IT — la sobriété numérique

Le numérique représente environ 4 % des émissions mondiales de gaz à effet de serre en 2026, et sa croissance continue. Réduire cette empreinte devient une exigence à la fois règlementaire (CSRD européen), économique (l'énergie coûte plus cher) et éthique.

Mesurer
Carbon footprint tools
Les hyperscalers exposent les émissions par service : AWS Customer Carbon Footprint Tool, Azure Sustainability Calculator, Google Cloud Carbon Footprint. Précision variable, surtout utile pour suivre les tendances.
Développer sobre
Green coding
Frugal en ressources dès la conception : images Docker minimales, cache agressif, code optimisé, requêtes SQL efficaces. Green Software Foundation propose un référentiel de pratiques.
Choisir sa région
Region picking
Toutes les régions cloud n'ont pas le même mix énergétique. Une région avec beaucoup de nucléaire ou d'hydraulique émet beaucoup moins que du charbon. France et Suède sont typiquement les meilleures options en Europe.
Matériel
Hardware reconditionné
Acheter du matériel d'occasion ou prolonger la vie de l'existant réduit l'empreinte de fabrication (très significative). Orange s'est donné l'objectif de 15 % de matériel recyclé sur ses déploiements serveurs, selon leurs communications publiques.
Section 10 — Focus Orange

L'infrastructure Orange France

À partir d'ici, on s'appuie uniquement sur des sources publiques : articles de presse spécialisée, site open source d'Orange, conférences techniques, communiqués Orange Business. Ta réalité interne peut différer.

L'héritage OpenStack

Orange est un des plus gros utilisateurs et contributeurs d'OpenStack dans le monde télécom. L'histoire commence en 2012 avec Cloudwatt, offre IaaS souveraine française montée dans le cadre du projet Andromède (Grand Emprunt), deuxième entreprise européenne à intégrer la Fondation OpenStack. Cloudwatt devient filiale d'Orange à 100 % en 2015, fusionne avec Orange Business en 2016, puis ferme en janvier 2020, les clients étant invités à migrer vers Flexible Engine.

Flexible Engine, l'offre IaaS publique d'Orange Business, est basée sur OpenStack et disponible en France et à l'international. Elle reste active en 2026.

Côté open source, Orange contribue activement : Orange dirige le projet OpenStack BGPVPN dans Neutron, facilitant l'interconnexion de réseaux Neutron à des IPVPN, et a porté le projet Moon, système de gestion de sécurité orientée utilisateur pour les services cloud OpenStack, accepté par le Technical Steering Committee d'OPNFV.

Les SDN POPs

Orange exploite un réseau mondial de points de présence SDN (Software Defined Networking Points of Presence) — environ une cinquantaine de localités tout autour du globe, avec un objectif de 75 d'ici à 2026. Ces datacenters font bien plus qu'interconnecter : ils exécutent plus d'une trentaine de services télécoms liés au routage, assurent la continuité et l'étanchéité de réseaux privés virtuels, maintiennent les connexions SD-WAN, IP/MPLS, FTTO, et exécutent toutes les couches de cybersécurité.

Historiquement, Orange, comme la plupart des télécoms, motorisait ses datacenters opérateur avec OpenStack, qui s'est particulièrement imposé chez les télécoms pour virtualiser sur des serveurs génériques des services (en particulier dans la téléphonie mobile) qui étaient auparavant livrés dans des boîtes noires par leurs fournisseurs.

La modernisation OINIS 2023-2026

C'est probablement le chantier infra le plus intéressant côté Orange ces dernières années. OINIS a lancé à l'été 2023 un appel d'offres pour moderniser ses infrastructures OpenStack, avec la capacité d'exécuter à la fois des VM (VNF, Virtual Network Functions) et des containers (CNF, Cloud-native Network Functions).

Parmi les dix candidats et les quatre finalistes, Red Hat a proposé une approche originale : alors que tous les autres fournisseurs proposaient un hyperviseur pour exécuter les VM, puis ajoutaient Kubernetes dans ces VM pour orchestrer les containers, Red Hat a inversé l'approche — traiter VM et containers au même niveau via Kubernetes. C'est KubeVirt — une extension qui permet à Kubernetes d'orchestrer directement des VMs comme si c'étaient des pods. Orange a retenu cette approche.

Autre contrainte marquante : un objectif de durabilité interne. D'après Bess Diop cité dans LeMagIT, « sur 100 serveurs déployés, 15 doivent être du matériel recyclé ». Un KPI Green IT concret, directement contractualisé dans les appels d'offres.

Ce que ça veut dire concrètement

Orange est en train de refondre son backbone réseau sur une stack qui combine le meilleur des deux mondes : la maturité OpenStack pour la virtualisation télécom historique, et la flexibilité de Kubernetes pour les nouvelles fonctions cloud-native. C'est une approche techniquement à la pointe, rare dans l'industrie, et c'est un bon point de référence quand tu croises VNF, CNF, ou KubeVirt dans des documents internes.

L'écosystème Orange Business

Orange Business Services (qu'on croise maintenant souvent sous Orange Business tout court) opère l'offre cloud et services numériques pour les entreprises. D'après leur propre documentation publique, leur offre couvre :

  • Flexible Engine — IaaS public souverain basé OpenStack
  • Des services managés pour Docker et Kubernetes, avec partenariats hyperscalers (AWS, Azure, GCP)
  • Du private cloud sur infrastructures dédiées
  • De la colocation dans leurs datacenters
  • Des offres verticales : Big Data / Data Lake, IoT, AI/ML, CNApps (cloud-native applications)
  • La chaîne outillée DevOps pour accompagner les clients dans leurs pratiques agiles

Glossaire express

ACID
Atomicity, Consistency, Isolation, Durability. Les quatre garanties des transactions dans une base relationnelle.
AZ
Availability Zone. Un datacenter ou petit groupe de datacenters à l'intérieur d'une région cloud.
Backstage
Portail développeur open source créé par Spotify, devenu standard de fait pour les IDP.
Bleu
Co-entreprise Orange + Capgemini pour opérer des services Microsoft sous droit français.
CDN
Content Delivery Network. Cache géo-distribué pour servir du contenu statique près des utilisateurs.
Cinder
Le service de stockage bloc d'OpenStack. Équivalent d'AWS EBS.
CNF
Cloud-native Network Function. Version conteneurisée d'une fonction réseau télécom. Successeur du VNF.
Crossplane
Outil qui expose les ressources cloud (AWS, GCP, Azure) comme des objets Kubernetes. IaC via l'API K8s.
FaaS
Function as a Service. Serverless — AWS Lambda, Cloud Functions.
FinOps
Discipline de pilotage et d'optimisation des coûts cloud.
Flexible Engine
L'offre IaaS public d'Orange Business Services, basée sur OpenStack.
Gaia-X
Initiative franco-allemande de cadre de labellisation des offres cloud européennes.
Golden path
Parcours pré-configuré et opinion-né proposé aux développeurs par l'équipe plateforme.
HSM
Hardware Security Module. Équipement dédié au stockage sécurisé de clés cryptographiques.
Hyperviseur
Logiciel qui partage un serveur physique en plusieurs machines virtuelles isolées.
IDP
Internal Developer Platform. Plateforme interne construite par l'équipe platform engineering pour les équipes produit.
IOPS
Input/Output Operations Per Second. Mesure de performance d'un stockage bloc.
KubeVirt
Extension Kubernetes permettant d'orchestrer des VMs comme des pods. Choix de Red Hat pour Orange OINIS.
LB (L4/L7)
Load Balancer niveau 4 (TCP/UDP) ou niveau 7 (HTTP). Le L7 fait du routage par URL, cookie, header.
MinIO
Stockage objet open source compatible S3, très utilisé en on-premise.
NAT Gateway
Permet aux ressources d'un subnet privé d'accéder à Internet sans être directement exposées.
Neutron
Le service réseau d'OpenStack. Gère VPCs, subnets, routers, floating IPs.
Nova
Le service compute d'OpenStack. Orchestre les machines virtuelles. Équivalent d'AWS EC2.
OCI
Open Container Initiative. Spécification ouverte du format et du runtime des containers.
OINIS
Entité d'Orange gérant l'infrastructure réseau (International Networks & Infrastructure Services), à l'origine de la modernisation OpenStack+Kubernetes récente.
OLAP / OLTP
Analytical vs Transactional Processing. Deux modes d'utilisation des bases de données.
OpenShift
Distribution Kubernetes d'entreprise de Red Hat, très présente chez les opérateurs télécom.
OpenStack
Plateforme open source d'infrastructure cloud. Modulaire (Nova, Neutron, Cinder, Swift, Keystone…). Socle historique d'Orange.
PoP
Point of Presence. Un site où un opérateur offre une connectivité réseau. Orange exploite plusieurs dizaines de SDN PoPs.
S3
Le service de stockage objet d'AWS, dont l'API est devenue le standard de fait partout.
SDN
Software-Defined Networking. Réseau où la configuration et le contrôle sont logiciels, découplés du matériel.
SD-WAN
Software-Defined Wide Area Network. Réseau étendu dont les politiques de routage sont pilotées par logiciel. Pilier des services Orange Business.
Serverless
Modèle où le provider gère tout le runtime et facture à l'exécution. FaaS ou conteneurs serverless.
Service mesh
Infrastructure de communication entre microservices. Istio, Linkerd.
Sovereign cloud
Cloud opéré sous droit local, hors juridiction étrangère (Patriot Act, CLOUD Act).
Subnet
Sous-réseau à l'intérieur d'un VPC, typiquement rattaché à une AZ.
Swift
Le service de stockage objet d'OpenStack. Équivalent d'AWS S3.
TSDB
Time-Series Database. Base optimisée pour les données horodatées.
VNF
Virtual Network Function. Version virtualisée d'une fonction réseau télécom (firewall, router, BRAS). Tourne dans une VM.
VPC
Virtual Private Cloud. Réseau isolé dans un cloud public.
