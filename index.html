Section 01 — Contexte

Le problème que ça résout

Avant Terraform, créer une infrastructure cloud ressemblait à une longue séance de clic-clic dans une console web. On crée un serveur, on attribue une IP, on configure un pare-feu, on branche une base de données… À la main. À chaque environnement (dev, recette, prod), tout est à refaire. Et personne ne se souvient exactement de ce qui a été fait.

L'Infrastructure as Code (IaC) inverse complètement la logique : au lieu de cliquer, on écrit ce qu'on veut, dans un fichier texte. Un outil se charge ensuite de créer les ressources réelles pour qu'elles correspondent exactement à la description.

Figure 1Du clic manuel vers l'infrastructure décrite dans du code

Les avantages sont nombreux et concrets : on peut versionner son infrastructure dans Git, faire des code reviews, répliquer un environnement à l'identique en quelques minutes, et garder une trace complète de qui a changé quoi et quand.

À retenir

Terraform n'est qu'un des outils d'IaC. Sa particularité : il est déclaratif (on décrit l'état souhaité, pas les étapes pour y arriver) et multi-cloud (AWS, Azure, GCP, OVH, Scaleway, Kubernetes… via le même langage).

Section 02 — Définition

Qu'est-ce que Terraform ?

Terraform est un outil open source créé par HashiCorp en 2014. Il permet de définir, provisionner et gérer des infrastructures cloud à partir de fichiers texte écrits dans un langage maison appelé HCL (HashiCorp Configuration Language).

Concrètement, c'est un binaire en ligne de commande — un seul exécutable que tu installes sur ton poste ou sur un serveur de CI. Tu lui donnes des fichiers .tf, il discute avec les API du cloud, et il fait apparaître (ou disparaître) les ressources demandées.

L'idée centrale en une phrase

Tu décris ce que tu veux. Terraform calcule la différence avec ce qui existe, et fait le strict nécessaire pour combler l'écart.

Cette logique de convergence vers un état cible est ce qui distingue Terraform d'un script bash classique. Si tu lances terraform apply dix fois de suite avec la même configuration, il ne se passera rien après le premier appel : l'état réel correspond déjà à l'état désiré.

Note historique

En 2023, HashiCorp a changé la licence de Terraform (passage de MPL à BSL), ce qui a déclenché la création d'un fork open source 100 % communautaire : OpenTofu, hébergé par la Linux Foundation. Les deux outils sont aujourd'hui largement compatibles. On y reviendra.

Section 03 — Cycle de vie

Comment ça fonctionne

Le quotidien d'une utilisatrice de Terraform tient en quatre commandes principales : init, plan, apply, et de temps en temps destroy. Ce cycle est la base à mémoriser.

Figure 2Le cycle de vie principal : write → init → plan → apply (et un destroy en sortie)

Détail de chaque commande

terraform init

Initialise le dossier de travail. Téléchargement des providers nécessaires (les plugins qui parlent aux différents clouds), configuration du backend où sera stocké le state. À lancer une fois au début, et à chaque ajout de provider.

terraform plan

Affiche un diff entre l'état actuel de l'infrastructure et l'état décrit dans tes fichiers. Une étape capitale : tu vois exactement ce qui sera créé (+), modifié (~) ou supprimé (-) avant que rien ne se passe pour de vrai.

terraform apply

Exécute le plan. C'est l'étape qui parle aux API du cloud et provisionne réellement les ressources. Une confirmation est demandée par défaut (sauf en CI/CD avec le flag -auto-approve).

terraform destroy

Supprime tout ce que Terraform gère pour ce projet. Très utile en environnement de test ; à manier avec énormément de prudence en production.

Astuce débutante

Toujours, toujours lire la sortie de terraform plan avant d'appliquer. C'est la dernière barrière entre toi et un DROP TABLE involontaire de ton infrastructure.

Section 04 — Vocabulaire

Les concepts clés

Sept mots à intégrer pour comprendre n'importe quel projet Terraform. Aucun n'est compliqué isolément ; c'est leur combinaison qui fait la puissance de l'outil.

Concept 01
Provider
Plugin qui sait parler à une API précise (AWS, Azure, GCP, GitHub, Datadog, Cloudflare, OVH…). C'est le traducteur entre ton code et le service distant. Il en existe plus de 4 000 dans le registre public.
Concept 02
Resource
Une chose que Terraform doit gérer : un serveur EC2, un bucket S3, un enregistrement DNS, un repo GitHub… Chaque resource a un type (fourni par le provider) et un nom (choisi par toi).
Concept 03
Data source
Lecture seule. Permet d'aller chercher des informations existantes (l'ID d'une AMI, un VPC déjà créé, un secret) sans les gérer. Pratique pour s'intégrer à de l'existant.
Concept 04
Variable
Paramètre d'entrée de ton code. Permet de réutiliser la même configuration avec des valeurs différentes (région, taille d'instance, nom d'environnement…). Définie dans variables.tf, valorisée via .tfvars.
Concept 05
Output
Valeur de sortie. Une fois l'infra créée, expose des informations utiles (l'IP publique du serveur, l'URL d'une base…) à afficher en console ou à passer à un autre module.
Concept 06
Module
Un dossier de fichiers Terraform que tu peux réutiliser comme une fonction. Encapsule un pattern (un cluster Kubernetes, un site statique, un VPC complet) et l'expose avec quelques variables.
Concept 07
State
Le fichier dans lequel Terraform mémorise ce qu'il a créé. Crucial : c'est sa source de vérité pour calculer les diffs. Section dédiée plus bas.

Section 05 — Langage

La syntaxe HCL

HCL est conçu pour être lisible par un humain. Pas de virgules en fin de ligne, pas de point-virgules, une syntaxe à base de blocs { }. Si tu connais JSON ou YAML, tu seras à l'aise en cinq minutes.

Voici un exemple complet et commenté qui crée un bucket de stockage AWS :

main.tf HCL 
# 1. On déclare le provider — d'où vient la "magie"
terraform {
  required_providers = {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "eu-west-3"  # Paris
}

# 2. Une variable d'entrée — pour rendre le code réutilisable
variable "environment" {
  type        = string
  description = "dev, staging ou prod"
  default     = "dev"
}

# 3. La ressource elle-même
resource "aws_s3_bucket" "docs" {
  bucket = "orange-docs-${var.environment}"

  tags = {
    Project     = "documentation"
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# 4. Une sortie — pour récupérer une info utile après création
output "bucket_url" {
  value       = aws_s3_bucket.docs.bucket_domain_name
  description = "URL d'accès au bucket"
}

Quatre blocs, et tu as une infrastructure complète, paramétrable, et reproductible. Tu peux en faire dix copies pour dix équipes en changeant simplement la variable environment.

Anatomie d'un bloc resource

Figure 4La signature d'un bloc resource — trois éléments dans cet ordre, toujours

Pour faire référence à cette ressource ailleurs dans ton code, tu utilises la syntaxe aws_s3_bucket.docs.<attribut>. C'est aussi ce mécanisme qui crée implicitement les dépendances entre ressources : Terraform comprend tout seul que A doit être créée avant B si B référence un attribut de A.

Section 06 — Concept critique

Le State, pierre angulaire

Si tu ne dois retenir qu'une chose après lecture de ce guide : le state est l'élément le plus important — et le plus piégeux — de Terraform. Comprendre ce qu'il est, c'est comprendre Terraform.

À quoi sert-il ?

Quand tu écris resource "aws_s3_bucket" "docs", Terraform crée le bucket sur AWS. Mais comment, lors du prochain apply, va-t-il savoir que ce bucket précis est celui qu'il a créé, et non pas un autre ? Réponse : grâce au state.

Le state est un fichier JSON (par défaut terraform.tfstate) qui établit la correspondance entre les noms logiques de ton code (aws_s3_bucket.docs) et les identifiants réels côté cloud (arn:aws:s3:::orange-docs-prod).

Figure 5Le state est la mémoire qui relie ton code au monde réel

Pourquoi c'est un sujet sensible

Le state contient parfois des secrets en clair (mots de passe générés, clés d'API…). Il est aussi la source de vérité de Terraform : si tu le perds, Terraform pense que rien n'existe et essaiera de tout recréer. Si deux personnes le modifient en même temps, c'est la corruption garantie.

D'où la règle d'or quand tu travailles à plusieurs : ne jamais garder le state en local. On utilise un backend distant qui le stocke de façon centralisée, chiffrée, et avec un mécanisme de locking qui empêche deux personnes de l'éditer en même temps. Les options classiques :

  • S3 + DynamoDB sur AWS (le plus répandu)
  • Azure Blob Storage sur Azure
  • GCS sur Google Cloud
  • HCP Terraform ou Terraform Enterprise (l'offre managée de HashiCorp)
  • Spacelift, env0, Scalr… les plateformes tierces qui gèrent ça pour toi
Règle absolue

Ne commite jamais un fichier terraform.tfstate dans Git. Mets-le dans ton .gitignore dès le premier projet. C'est l'erreur numéro un des débutantes.

Section 07 — Outillage

L'écosystème

Terraform tout seul, c'est déjà puissant. Mais autour de lui s'est construit, en dix ans, tout un écosystème d'outils qui adressent les besoins concrets de la vie en équipe : organiser des centaines de modules, sécuriser le code, automatiser les déploiements, tester, gérer plusieurs environnements proprement…

Figure 6Cartographie de l'écosystème Terraform — six grandes familles d'outils

Les outils à connaître par catégorie

Fork open source
OpenTofu
Né en 2023 du changement de licence de Terraform. Drop-in replacement, même langage HCL, ajoute quelques features (variables dans le bloc backend, state encryption natif). Choix par défaut si tu démarres aujourd'hui en pur open source.
Wrapper / DRY
Terragrunt
Pour éviter le copier-coller entre dev/, staging/, prod/. Apporte des fichiers terragrunt.hcl qui factorisent la config commune et orchestrent les dépendances entre stacks.
CI/CD spécialisé
Atlantis
Bot self-hosted qui exécute plan automatiquement à l'ouverture d'une pull request, et apply sur commande dans la PR. Convient pour internaliser un workflow GitOps gratuitement.
Plateforme managée
HCP Terraform
L'offre cloud de HashiCorp (anciennement Terraform Cloud). Backend de state, exécution distante, policy-as-code via Sentinel, gestion de modules privés. Tier gratuit pour petites équipes.
Plateforme managée
Spacelift / env0 / Scalr
Concurrents de HCP Terraform, avec souvent une approche plus flexible (multi-stack, drift detection avancée, support OpenTofu et Pulumi). À considérer pour les équipes plateform qui gèrent du Terraform à grande échelle.
Sécurité / policy
Checkov · Trivy · tfsec
Scanners statiques qui inspectent ton code Terraform à la recherche de mauvaises configurations (bucket S3 public, security group ouvert, chiffrement absent…). Indispensable en CI dans tout projet sérieux.
Qualité de code
tflint
Linter spécifique à Terraform. Détecte les erreurs de syntaxe avancées, les valeurs invalides pour ton provider, les conventions non respectées. Pense ESLint mais pour HCL.
Tests
Terratest · terraform test
Pour tester réellement un module en provisionnant une infra éphémère et en vérifiant son comportement. terraform test est la solution native depuis la v1.6 ; Terratest reste plus puissant mais demande de coder en Go.
Documentation
terraform-docs
Génère automatiquement la doc d'un module (variables, outputs, providers utilisés) au format Markdown. À brancher en pre-commit pour avoir une doc toujours à jour.
Visualisation
InfraMap · Rover · Pluralith
Génèrent des schémas visuels de l'infrastructure à partir du state ou du plan. Utile pour les revues d'architecture et l'onboarding d'équipiers.
Coût
Infracost
Estime le coût mensuel d'un changement Terraform avant qu'il soit appliqué. Affichable directement dans une PR. Très apprécié des équipes qui veulent un garde-fou financier.
Génération
Terraformer
Le sens inverse : à partir d'une infra existante, génère le code Terraform et le state correspondants. Utile pour reprendre du legacy en main.
Section 08 — Comparatif

Terraform vs les alternatives

Terraform n'est pas le seul outil d'IaC. Il vit dans un paysage où plusieurs approches coexistent — chacune avec ses forces. Voici comment les situer :

Outil Approche Quand le choisir
OpenTofu Fork open source de Terraform — même langage, même écosystème, gouvernance Linux Foundation. Tu démarres aujourd'hui sans contrainte, ou tu veux te libérer de la BSL HashiCorp tout en gardant tes acquis HCL.
Pulumi IaC en vrais langages : TypeScript, Python, Go, C#. Pas de DSL. Ton équipe est composée de développeurs qui veulent boucler, tester, abstraire avec leurs outils habituels.
AWS CloudFormation Service natif AWS, YAML/JSON, géré côté AWS sans state local. Tu es 100 % AWS et tu veux un outil officiel, intégré à IAM et au support AWS.
AWS CDK / CDKTF Génération de templates CloudFormation (CDK) ou Terraform (CDKTF) depuis du code TS/Python. Tu veux le confort d'un vrai langage tout en gardant un moteur d'exécution standard derrière.
Pour-cloud (Bicep, Deployment Manager) Outils natifs d'Azure et GCP, similaires à CloudFormation. Mono-cloud assumé, intégration profonde avec le portail du fournisseur.
Ansible Outil de configuration management, plus orienté gestion d'OS et déploiement applicatif que provisioning d'infra. Tu configures des serveurs existants (paquets, fichiers, services). Souvent complémentaire à Terraform plutôt qu'un concurrent.
Crossplane Provisioning d'infra via Kubernetes : tes resources cloud deviennent des objets Kubernetes. Tu vis déjà 100 % dans Kubernetes et tu veux un seul plan de contrôle pour les workloads et l'infra.
Notre recommandation

Pour un débutant en 2026, le couple Terraform (ou OpenTofu) + un backend distant reste la voie la plus pédagogique : la communauté est immense, la documentation est partout, et les compétences se transfèrent dans toutes les entreprises.

Section 09 — Réflexes

Bonnes pratiques pour débuter

Voici dix règles qui t'éviteront 80 % des problèmes. Elles n'ont rien de sophistiqué — la plupart des incidents sérieux viennent de leur non-respect.

  • Mets ton state en backend distant dès le premier projet S3, GCS, Azure Blob, ou une plateforme SaaS. Ne reste jamais en local au-delà d'un POC personnel.
  • Versionne tout dans Git, mais pas le state Tes .tf dans Git, ton .tfstate jamais. Mets *.tfstate* dans le .gitignore immédiatement.
  • Pin tes versions de provider Utilise required_providers avec une contrainte précise (~> 5.0). Une mise à jour silencieuse peut casser ton plan.
  • Lis le plan en entier avant d'apply Surtout les lignes -/+ qui signifient destruction puis recréation. C'est là qu'on perd des bases de données.
  • Sépare tes environnements physiquement Un dossier (et idéalement un state) par environnement : dev/, staging/, prod/. Ou un workspace dédié par env.
  • Modularise quand un pattern se répète À partir du moment où tu copies-colles le même bloc deux fois, c'est qu'il faut un module. Pas avant.
  • Utilise le registre public registry.terraform.io regorge de modules officiels (AWS, Azure, GCP). Inutile de réinventer un VPC depuis zéro.
  • Branche un scanner de sécurité en CI Checkov ou Trivy dans la pipeline. Détecte les buckets ouverts, les SG trop permissifs, les volumes non chiffrés.
  • Active le format et le lint en pre-commit terraform fmt et tflint automatiquement avant chaque commit. Code propre, diffs lisibles.
  • Documente tes modules Variables et outputs avec une description. Génère le README avec terraform-docs. Tes futurs toi te remercieront.
Section 10 — Ressources

Pour aller plus loin

Tu as maintenant la carte mentale. Pour la pratique, voici un parcours d'apprentissage qui fonctionne bien :

Étape 1 — Faire son premier apply

Installe Terraform (ou OpenTofu), crée un compte gratuit chez un cloud provider, et suis le tutoriel officiel « Get Started » sur developer.hashicorp.com. Compte 1 à 2 heures pour le faire de bout en bout. Tu verras une vraie ressource apparaître dans la console — c'est le moment où ça clique.

Étape 2 — Construire un projet personnel

Choisis un projet simple : un bucket S3 pour héberger un site statique, une instance EC2 avec un nom de domaine, une base de données PostgreSQL managée. L'objectif : passer par init, plan, apply, faire évoluer la config, refaire un apply, puis destroy.

Étape 3 — Apprendre les modules

Refactorise ton projet en extrayant un module. Puis utilise un module du registre public (par exemple terraform-aws-modules/vpc/aws) pour voir comment les autres font.

Étape 4 — Mettre en place une CI

Branche ton repo sur GitHub Actions ou GitLab CI. Fais tourner terraform fmt -check, terraform validate, tflint, checkov, et un terraform plan à chaque PR. C'est l'étape qui te fait passer de l'amateur au pro.

Ressources de référence

  • Documentation OpenTofu — opentofu.org/docs
  • Registre des providers et modules — registry.terraform.io
  • Terraform Up & Running de Yevgeniy Brikman — la référence livre, en cours de mise à jour régulière

Mot de la fin

Terraform a la réputation d'être hostile au début. C'est faux : il est juste strict. Une fois passé le moment où tu comprends le rôle du state et le cycle plan/apply, tout devient mécanique. La courbe d'apprentissage initiale est de 2 à 3 jours pour les bases, quelques semaines pour être à l'aise sur un vrai projet, quelques mois pour maîtriser l'écosystème entier. Bon voyage.
